# Claude Code 进阶 · 5月20日

## 今日主题：CLAUDE.md 高级工程 —— 多文件规则、条件触发与上下文压缩

你的 `CLAUDE.md` 已经定义了仓库结构、Agent 路由、输出偏好——但它可能面临一个隐患：**越长越好吗？** Claude Code 每次启动都会完整加载 CLAUDE.md，一个 5KB 的规则文件消耗约 1500 tokens。如果你的规则超过 10KB，每次对话光读规则就占掉 30% 的上下文窗口。

今天学会**规则工程**——用多文件拆分、条件触发、环境针对性规则把 CLAUDE.md 从「大杂烩」变成「智能路由系统」。

## 操作步骤

### 第1步：理解 Claude Code 规则加载优先级

Claude Code 从以下位置（优先级从高到低）合并规则：

| 优先级 | 路径 | 作用范围 |
|--------|------|---------|
| 1 (最高) | `项目根/.claude/rules/*.md` | 按文件名排序后**全部加载**，每个文件独立 |
| 2 | `项目根/CLAUDE.md` | 全局规则，整个仓库有效 |
| 3 | `项目根/.claude/CLAUDE.md` | 功能等同 CLAUDE.md，二选一即可 |
| 4 | `~/.claude/CLAUDE.md` | 全局用户级别规则，所有项目生效 |
| 5 | `--rules-file` 参数 | 显式指定的规则文件 |

**关键洞察**：`~/.claude/CLAUDE.md`（用户级）和 `.claude/rules/*.md`（分文件规则）是两个被低估的进阶能力——前者让你跨项目统一行为，后者让你按主题拆分规则。

### 第2步：搭建分层规则体系

**第一步：创建用户级全局规则**

`~/.claude/CLAUDE.md` 中的规则对**所有项目**生效。适合放通用行为规范：

```bash
mkdir -p ~/.claude
```

写入 `~/.claude/CLAUDE.md`：

```markdown
## 通用行为规范

### 输出语言
- 默认使用中文，专业术语保留英文缩写

### 文件安全
- 对被 `gitignore` 文件的操作先询问确认
- 批量修改超过 5 个文件时，先展示修改计划，等我确认再执行
- 不删除任何文件，除非我明确说了「删除」

### 工具使用
- 优先使用 Read 工具查看文件内容，而非 grep/find
- 写入文件前先检查路径是否存在
- 执行 shell 命令时始终使用 `workdir` 明确工作目录

### 跨项目通用约束
- 不在 `node_modules/`、`.git/`、`venv/` 目录中新建或修改文件
- 不生成 `.env` 文件——各项目的密钥用 `--env-file` 或环境变量注入
```

**第二步：将大规则拆分为 `.claude/rules/` 分片**

这是核心技巧——把 Carl 仓库的大 CLAUDE.md 按主题拆分成独立的规则文件，Claude Code 会自动加载**全部** `.claude/rules/*.md`：

```bash
# 在项目根目录
cd ~/wiki
mkdir -p .claude/rules

# 查看当前 CLAUDE.md 大小
wc -c CLAUDE.md
# 如果超过 3000 字节，强烈建议拆分
```

拆分为4个文件：

```bash
# 文件1：仓库架构与目录说明
cat > .claude/rules/01-repo-structure.md << 'RULES'
# 仓库结构

## 两大核心文件夹
- `ACT笔记-深度思考/` — 个人思考与项目执行（六层收件箱→归档）
- `LLM Wiki知识应用/` — 外部知识采集与萃取（raw→entities→concepts/reports）

## 收件箱
- `收件箱：同步助手/` — 同步操作日志（含 Cloude Code学习/、企业出海日报/）
- `收件箱：需求模拟/` — 每日客户分析报告
RULES

# 文件2：Agent 体系与路由（只保留路由逻辑，省略每个 Agent 的完整定义）
cat > .claude/rules/02-agent-routing.md << 'RULES'
# Agent 路由规则

三层 Agent：orchestrator（编排器）→ intel-expert / diagnose-client / consult-client

路由逻辑：用户说英语培训/管理咨询 → 走编排器；用户说 /diagnose-client → 直接路由。
编排器检测到情报查询信号时，优先路由到 intel-expert。
RULES

# 文件3：LLM Wiki 运行机制
cat > .claude/rules/03-wiki-operations.md << 'RULES'
# LLM Wiki 操作

四层：Ingest（摄入）→ Query（查询）→ Archive（归档）→ Lint（审计）

默认：所有产出 → wiki 优先。Wiki 是唯一真相源，不入库不算完成。

ACT ↔ Wiki 互通：项目启动时关联 wiki 素材，完成后从 ACT 萃取出 entity。
RULES

# 文件4：输出偏好与约束
cat > .claude/rules/04-output-style.md << 'RULES'
# 输出风格

- 中文为主，术语保留英文缩写
- 复杂逻辑用表格 + 结构化清单
- 每个建议指向可执行的动作或判断标准
- 不确定时只说「需要查证」，不编造
- 目标先行：目标→策略→流程→分工
- 资源前置：做事前先看知识库
RULES
```

拆分后，Claude Code 启动时会自动加载 `01-repo-structure.md`、`02-agent-routing.md`、`03-wiki-operations.md`、`04-output-style.md` 四个文件。**按序号控制优先级**（数字小先加载），**按文件隔离职责**，不再是一锅粥。

### 第3步：利用 `CLAUDE.md` 的声明式限制保护关键操作

在 `CLAUDE.md` 或 `~/.claude/CLAUDE.md` 中写限制性规则来防止误操作：

```markdown
## ⚠️ 安全保护规则（不可绕过）

### 批量写入保护
- 一次操作超过 10 个文件时，必须分两步：
  1. 先展示完整的改动计划（文件名 + 改动类型 + 预估影响）
  2. 等我回复「执行」后才进行写入
- 对 `ACT笔记-深度思考/` 下的文件做 `mv` 或 `rm` 前必须双重确认

### 网络请求限制
- 不访问内网地址（10.x.x.x、172.16-31.x.x、192.168.x.x）
- 不向生产环境的数据库写入测试数据

### Git 操作保护
- 在 main 分支上执行 `git commit` 前必须问我「确定直接提交到 main？建议先创建分支」
- 不执行 `git push --force`
```

### 第4步：使用 `--rules-file` 针对不同场景加载专用规则

这是最有用的进阶技巧——不同任务加载不同的规则集：

```bash
# 场景1：批量处理 Obsidian 收件箱 → 加载收件箱规则
claude -p "处理这批文件" \
  --rules-file ~/wiki/.claude/rules/05-inbox-processing.md

# 场景2：生成客户方案 → 加载方案设计规则
claude -p "为中信生成培训方案" \
  --rules-file ~/projects/客户方案/.claude/proposal-rules.md

# 场景3：代码审查 → 加载审查规则（简小精悍）
claude --allowedTools "Read" -p "审查这个 PR" \
  --rules-file ~/wiki/.claude/rules/06-code-review.md
```

**进阶模式：环境感知的自动化规则加载**

```bash
# ~/.bashrc 或 ~/.zshrc 中添加
cc-task() {
  local task_type="$1"
  shift
  local rules_file=""
  case "$task_type" in
    inbox)    rules_file="$HOME/wiki/.claude/rules/05-inbox-processing.md" ;;
    proposal) rules_file="$HOME/projects/.claude/proposal-rules.md" ;;
    review)   rules_file="$HOME/wiki/.claude/rules/06-code-review.md" ;;
    weekly)   rules_file="$HOME/wiki/.claude/rules/07-weekly-review.md" ;;
    *)        rules_file="" ;;
  esac
  if [ -n "$rules_file" ]; then
    claude -p "$*" --rules-file "$rules_file"
  else
    claude -p "$*"
  fi
}

# 用法
# cc-task inbox "处理今日收件箱新增笔记"
# cc-task proposal "生成中信培训方案 v2"
# cc-task review "审查我未提交的变更"
# cc-task weekly "生成本周工作复盘"
```

### 第5步：利用 `--allowedTools` 精细控制权限

在每个规则文件或 `CLAUDE.md` 中配合 `allowedTools` 做微调：

```markdown
# 收件箱处理规则（05-inbox-processing.md）
## 工具限制
此任务只能使用：
- Read（读取文件）
- Edit（编辑文件 frontmatter 和内容）
- Write（新建文件）
- Bash（需要 grep/ls 辅助）

禁止使用：
- Execute（运行代码）
- Web（网络请求）
- Computer（系统级别操作）
```

配合命令行：

```bash
claude --allowedTools "Read,Edit,Write" -p "批量处理收件箱"
```

### 第6步：规则文件瘦身检查表

你的 CLAUDE.md 是否需要拆分？对照自查：

| 问题 | 建议处理 |
|------|---------|
| CLAUDE.md 超过 5KB | 拆分到 `.claude/rules/` |
| 包含 3 个以上不同职责的区块 | 按职责拆分 |
| 有超过 20 条具体规则 | 整理为模板 + 精简版 |
| 所有 Agent 的完整定义都在一个文件里 | 每个 Agent 独立文件 |
| 包含只对特定任务有用的规则（如 Git LFS） | 移到对应的专用规则文件 |

## 在Carl工作中的应用

### 场景：不同任务加载不同规则集

你每天切换多个角色——每个角色需要 Claude Code 有不同的行为：

```bash
# 📌 场景A：知识管理（「知识虾」模式）
# 规则：允许批量操作、写入 wiki、网络搜索
cc-task inbox "扫描1-收件箱/灵感记/，归类并打标签，将提炼内容摄入 LLM Wiki"

# 📌 场景B：客户方案设计（「方案虾」模式）
# 规则：严格禁止网络请求，只读客户信息，专注文档生成
cc-task proposal "基于客户需求生成一份R.I.S.E.跨文化培训方案"

# 📌 场景C：情报分析（「情报虾」模式）
# 规则：只读分析，禁止写入，输出结构化报告
claude --allowedTools "Read,Execute" \
  --rules-file ~/wiki/.claude/rules/08-intel-analysis.md \
  -p "分析这周企业出海日报信号，输出竞争格局报告"
```

### 场景：拆分后的 CLAUDE.md 对比效果

| 维度 | 原来（大 CLAUDE.md） | 拆分后（.claude/rules/） |
|------|---------------------|------------------------|
| 文件大小 | ~5KB | 每个 0.5-1.5KB |
| 每次启动 Token 消耗 | ~1500 | ~500-800（按需加载） |
| 修改影响范围 | 改一处影响全部 | 只改对应规则文件 |
| 不同任务差异化 | 无（一锅粥） | 通过 `--rules-file` 切换 |
| 维护清晰度 | 滚动到底部找规则 | 文件即职责，一目了然 |

完成后你就掌握了「规则工程」的核心思路——下次再遇到 Claude Code 行为不符合预期的情况，不再往 CLAUDE.md 底部追加一行，而是新建一个规则文件或调整 `--rules-file`。**规则即代码——可拆分、可复用、可测试。**

## 注意事项

1. **`.claude/rules/` 加载全部文件**——Claude Code 会按文件名字母序加载所有 `.md` 文件。如果想控制加载顺序，文件名加前缀序号（`01-`、`02-`）。但 Claude Code 的规则优先级是按声明顺序处理，**后加载的规则可能覆盖前一个**——所以把最重要的规则放在序号最小的文件中
2. **不要硬删除主 CLAUDE.md**——迁移过程中主 CLAUDE.md 要保持，否则 `.claude/rules/` 文件加载完发现主文件不见了，Claude Code 会丢失项目上下文。推荐双轨运行一段时间：主 CLAUDE.md 只保留最核心的 3-5 条规则，详细内容移到 `.claude/rules/`
3. **`~/.claude/CLAUDE.md` 的副作用**——用户级规则对**所有项目**生效。如果你在一个与工作无关的仓库（如个人 side project）中运行 Claude Code，全局规则可能导致奇怪的行为。建议用户级规则只放最通用的约束（输出语言、安全规则）
4. **`--rules-file` 和 `.claude/rules/` 的关系**——`--rules-file` 指定的文件会**额外**加载，不会替代 `.claude/rules/` 的内容。如果你想只加载指定规则而不加载项目规则，目前没有直接方式，只能用 `--allowedTools` 做约束
5. **规则长度与速度的 tradeoff**——虽然拆分后每次加载的规则变少，但 Claude Code 仍会读取所有 `.claude/rules/*.md` 文件。10 个小型文件 vs 1 个大型文件，加载速度差异很小（毫秒级），但 Token 节省显著
6. **避免规则冲突**——当多个规则文件对同一行为有不同要求时，后加载的规则优先。保持规则文件的职责正交（不重叠），冲突时用序号优先级明确
7. **CLAUDE.md 中使用 YAML frontmatter**——Claude Code 规则文件支持可选的 YAML 前导元数据，可以用来标注模型偏好、最大输出 token 等，但注意 frontmatter 内容不参与规则匹配
