# Claude Code 进阶 · 5月14日

## 今日主题：CLAUDE.md 高级模式 —— 从"人物简介"到"指令系统"

你已经有了一个基础 CLAUDE.md（wiki/CLAUDE.md），里面写了你是谁、你的工作原则和输出偏好。但这只是冰山一角。Claude Code 的规则文件支持**分层覆盖、条件指令、参考链接和工具偏好声明**——今天把它升级成真正的指令系统。

核心原理：Claude Code 会从当前工作目录向上查找 `.claude.md`（项目级）和 `CLAUDE.md`（根级），并**合并**所有匹配的规则文件。这意味着你可以：

1. **根级 CLAUDE.md**：放全局行为原则（你已有的"不杜撰、不空话"原则）
2. **子目录 .claude.md**：放特定上下文的任务规则（例如 `projects/` 目录下放项目管理指令，`wiki/` 下放知识库操作指令）

## 操作步骤

### 1. 创建子目录级 .claude.md

在 Obsidian 知识库的不同区域创建专用规则文件：

```bash
# 知识库操作专用规则
cat > ~/wiki/LLM\ Wiki知识应用/.claude.md << 'RULES'
# LLM Wiki 知识库操作规则

## 知识萃取标准
当你被要求从素材中萃取知识时，按以下结构输出：
1. **核心概念** — 一句话定义（20字内）
2. **适用场景** — 这个知识在什么情境下有用
3. **操作清单** — 3-5个可执行步骤
4. **关联知识点** — 知识图谱中的相邻节点
5. **反例/陷阱** — 常见误解或错误做法

## 文件命名规范
- concepts: `concepts/英文短名.md`
- entities: `entities/组织名.md`
- methods: `methods/方法论英文名.md`
- 禁止中文文件名（除了索引页）
RULES

# 项目管理专用规则
cat > ~/wiki/ACT笔记-深度思考/.claude.md << 'RULES'
# ACT笔记 - 项目管理操作规则

## 每日归档流程
当你看到"日归档"指令时：
1. 检查 ACT笔记-深度思考/日日志/ 中昨天的文件
2. 检查 ACT笔记-深度思考/Action/ 中标记为"待归档"的项目
3. 将已完成项移动到 ACT笔记-深度思考/归档/ 对应月份文件夹
4. 更新相关项目卡片的状态
5. 生成归档摘要报告

## 项目文档模板
- 项目立项：包含背景、目标、里程碑、风险矩阵
- 项目复盘：包含数据回顾、亮点、改进点、沉淀资产
- 所有项目文档第一行必须是 `# 项目名称 | 项目编号 | 状态`
RULES
```

### 2. 升级根级 CLAUDE.md —— 加入工具偏好和条件块

```bash
# 追加到现有 ~/wiki/CLAUDE.md
cat >> ~/wiki/CLAUDE.md << 'ADDITIONS'

## 工具偏好
- 批量文件操作优先用终端命令（find/xargs/sed），而非逐文件编辑
- 知识库查询优先走 GBrain MCP（mcp_gbrain_query），其次才是本地文件搜索
- 生成 markdown 表格前，先考虑是否需要转为结构化清单（Carl 偏好清单 > 表格）
- 不要用 `ls` 和 `cat`，用 `search_files` 和 `read_file`

## 自动行为规则
当用户提到以下关键词时，自动触发对应流程：
- "进度"/"进展" → 执行 system-status-report 技能
- "深度提取"[something] → 执行 deep-extraction-methodology 技能
- "查知识"[keyword] → 执行 knowledge-quick-search 技能
- "帮我深挖"[topic] → 使用 GBrain think + traverse_graph 做知识图谱分析

## 代码/命令执行安全规则
- 涉及 rm -rf / git push --force / chmod -R 777 的命令，必须先列出要执行的命令，得到确认后再执行
- 批量修改文件前，先做 dry-run 预览改动
- 涉及客户数据的操作，先确认是否有备份
ADDITIONS
```

## 在Carl工作中的应用

**场景：你跟团队成员共享 Obsidian 知识库，每个人都在用 Claude Code 操作不同模块。**

团队成员小王负责项目管理，他打开 `ACT笔记-深度思考/` 目录下的文件时，Claude Code 自动加载那里的 `.claude.md`，里面定义了项目文档的固定模板、归档流程和日报格式。他只要说"写一个当升科技项目周报"，Claude Code 就自动按 SOP 生成了——不需要每次数模板结构。

你本人负责知识库建设，进入 `LLM Wiki知识应用/` 目录，Claude Code 自动知道"萃取知识必须按5段结构输出、中文文件名不行"。即使你只说"帮我深挖这篇跨文化沟通的文章"，输出质量也稳定统一。

**效果：团队5个人各管一块，每条指令都被各自的 .claude.md 校准，不用口口相传SOP。**

## 今日练习 (5分钟)

打开终端，执行以下命令给你的 CLAUDE.md 加一个"自动行为规则"区块：

```bash
cat >> ~/wiki/CLAUDE.md << 'PRACTICE'

## 自动行为规则
当我说"写周报"时：
1. 读取当前周的所有日日志文件
2. 提取完成事项和待办事项
3. 按"目标→进展→问题→下周计划"格式生成周报
4. 保存到 ACT笔记-深度思考/周报/ 文件夹，文件名格式为 `周报-{ISO周}.md`
PRACTICE
```

然后确认追加成功：

```bash
tail -15 ~/wiki/CLAUDE.md
```

**完成了？** 你现在已经建立了**规则分层体系**——以后每进入一个子目录，Claude Code 就自动切换工作模式。这就是从"一个人用"到"团队用"的分水岭。

## 注意事项

- **文件命名大小写敏感：** `.claude.md`（小写）是项目级，`CLAUDE.md`（大写）是根级。不要搞混。
- **递归加载顺序：** Claude Code 从当前目录向上查找，最近的规则优先级最高。所以子目录的 `.claude.md` 可以覆盖根级的同名指令。
- **规则膨胀陷阱：** 单个 CLAUDE.md 不要超过 200 行。超过后 Claude Code 在上下文窗口中的"注意力稀释"会导致尾部规则被忽略。按目录拆分是正解。
- **不要放机密信息：** CLAUDE.md 通常提交到版本控制。API Key、密码、客户合同金额不要写在规则文件里。
- **.gitignore 策略：** 如果不想把 `.claude.md` 提交到 repo（比如仓库是公开的），在 `.gitignore` 中加入 `.claude.md`。建议至少把根级 `CLAUDE.md` 提交——这才是团队协作的价值所在。
