# Claude Code 进阶 · 5月18日

## 今日主题：Git 工作流增强 —— 自动 commit、PR 创建与代码审查

你已经用 Claude Code 操作本地文件，但每次改完还要手动 `git add → commit → push`，或者打开 GitHub 网页提 PR。Claude Code 原生支持 Git 操作——它能看到 git diff、自动生成 commit message、创建 PR，甚至做代码审查。今天学会把 Claude Code 变成你的 Git 工作站，告别命令行 git 手打。

核心原理：Claude Code 内置了 Git 感知——它通过 `git diff`、`git log`、`git status` 感知仓库状态，并能通过 GitHub CLI (`gh`) 执行 PR 操作。你不需要记住 git 命令，用自然语言描述意图即可。

## 操作步骤

### 第1步：配置 Git + GitHub CLI 环境

确保你的环境中有基本的 Git 凭证和 gh CLI：

```bash
# 1. 确认 Git 已配置
git config --global user.name "Carl"
git config --global user.email "carl@imorgan.com"

# 2. 安装/确认 GitHub CLI
# macOS:
# brew install gh
# Ubuntu/Debian:
# sudo apt install gh

# 3. 验证 gh 可用
gh --version

# 4. 登录 GitHub（一次性的 OAuth 流程）
gh auth login
```

### 第2步：用 Claude Code 做智能 Commit

这是最常用的模式——在 Claude Code 中修改文件后，直接让它处理 git 流程：

```
# 在 Claude Code 交互模式中，完成文件修改后，说：

"列出所有未暂存的变更，按文件分类（新增/修改/删除），
帮我生成一个有意义的 commit message，
然后执行 git add 和 git commit"

# Claude Code 会：
# 1. 运行 git status + git diff 了解全貌
# 2. 生成结构化的 commit message（按改动模块分组）
# 3. 执行 git add -A && git commit -m "..."
```

### 第3步：一键创建 PR（Pull Request）

当你完成了一个功能分支的工作，用 Claude Code 创建 PR：

```bash
# 在功能分支上运行
claude -p "
当前分支有什么变更？
帮我创建一个 PR：
- 标题提取核心改动
- 正文包含：背景、做了什么、关键改动清单、测试说明
- 添加标签 'enhancement'
- 指派给 review 组
"

# 如果是要提给上游仓库（fork 场景）
claude -p "
我已经在 fork 分支上修改完成。
帮我创建一个 PR 到上游仓库（owner/repo）的 main 分支，
标题和正文反映这次改动的业务价值，
正文以 '## 改动清单' 开头列出每个文件的改动意图
"
```

### 第4步：Claude Code 做代码/文档审查

团队协作场景——别人提了 PR，用 Claude Code 审查：

```bash
# 审查一个 PR（按 PR 编号）
claude -p "
审查 PR #42：
1. 列出这个 PR 改动了哪些文件、多少行
2. 检查是否存在安全问题（硬编码密钥、注入风险等）
3. 文档类变更：检查格式一致性、链接是否有效
4. 给出整体评价：approve / needs-work / block
"

# 或者审查当前工作目录的未提交变更
claude -p "
审查我当前的未提交变更：
1. 分析每个改动的意图
2. 检查是否有遗漏的 TODO 或调试代码
3. 检查命名一致性（slug 格式、标签大小写）
4. 有没有更好的实现方式？
"
```

### 第5步：批量 Git 操作自动化脚本

把常用模式封装成 shell 函数，直接调用：

```bash
# 加到 ~/.bashrc 或 ~/.zshrc
cc-review() {
  local target="${1:-.}"
  cd "$target" || return 1
  claude -p "
审查当前 git 仓库的未提交变更：
- 这是 Obsidian 知识库，关注：frontmatter 完整性、链接有效性、标签一致性
- 检查是否有二进制文件被误提交
- 列出所有需要注意的问题
- 如果一切正常，输出 ✅ 没问题
" --no-print-rules
}

cc-pr() {
  local title="${1:-Auto PR}"
  claude -p "
创建 PR：
标题: $title
正文要包含改动的业务背景和技术说明。
如果是文档类改动，在正文末尾加上截图指引。
"
}

# 用法
# cc-review ~/wiki
# cc-pr "更新R.I.S.E.课程体系大纲"
```

### 第6步：CLAUDE.md 中预设 Git 规则

在 `CLAUDE.md` 中声明 Git 行为偏好，让每次提交都自动规范：

```yaml
# 在 ~/wiki/CLAUDE.md 中新增
## Git 操作规则

### Commit Message 规范
当需要生成 commit message 时，按以下格式：
```
<type>(<scope>): <简短描述>

<详细说明（可选）>
```

类型(type)：feat（新内容）/ fix（修复）/ docs（文档）/ refactor（重构）/ chore（杂项）
范围(scope)：rise（课程体系）/ intel（情报）/ client（客户）/ infra（基础设施）/ wiki（知识库）

### 自动提交策略
- 每天的日归档完成后，自动执行一次 git add + commit
- commit message 格式：`chore(daily): 归档 $(date +%F)`
- 不自动 push——push 需要我确认

### PR 模板
创建 PR 时使用以下正文模板：
```
## 背景
[为什么要做这个改动]

## 改动清单
- [文件路径]: [改动说明]

## 验证
- [ ] 本地预览正常
- [ ] 链接无断链
- [ ] frontmatter 完整
```

### 忽略文件
- 不要提交 `.obsidian/` 下的工作区配置（workspace.json、workspace-mobile.json）
- 不要提交 `_processed/` 下的处理记录
- 如果 `.gitignore` 中缺少以上项，提醒我添加
```

cd ~/wiki

# 第一步：看看有多少未提交的改动

# 第二步：让 Claude Code 生成一次智能提交
# （进入交互模式后执行：）

## 注意事项

1. **gh CLI 是必需的**——Claude Code 的 PR 创建功能通过 GitHub CLI (`gh`) 实现。没有安装 gh 的话，只能做本地 git 操作（commit、diff、log），无法创建 PR。安装: `brew install gh` 或 `sudo apt install gh`
2. **谨慎使用自动 push**——不要在 CLAUDE.md 中写 `自动 push`。先 commit 再手动确认 push，防止推送了不完整的改动。尤其是在多人协作的仓库中
3. **大文件提交警告**——Claude Code 不会主动检查文件大小。如果知识库中有超过 50MB 的附件（PDF、图片），git commit 前先确认 `.gitignore` 是否排除了这些文件。提示 Claude Code "检查是否有大文件被误包含"
4. **commit message 一致性**——多人协作时，commit message 风格混乱。把 Git 行为规则写进 CLAUDE.md 后，所有人在 Claude Code 中生成的 commit 都自动对齐格式
5. **`--no-print-rules` 加速**——在批量脚本中加 `--no-print-rules` 跳过打印 CLAUDE.md 内容，减少每次调用的 Token 消耗，批量操作快 30%
6. **Git LFS 支持有限**——如果仓库使用 Git LFS 管理大文件（设计稿、录音等），Claude Code 不会自动识别 LFS 跟踪的文件。需要先手动 `git lfs track` 再让 Claude Code 操作
7. **合并冲突处理**——Claude Code 能检测到合并冲突，但解决复杂冲突（尤其是跨多文件的冲突）时可能遗漏。建议：让 Claude Code 列出冲突文件，自己逐文件确认解决后再让它 commit
