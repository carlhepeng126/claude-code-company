### **🦞 核心理念：养虾而非租虾**

作者提出核心观点：**90%的人用OpenClaw只是在"租"一个AI，而非"养"一个伙伴**。OpenClaw出厂时是"三无产品"（无身份、无记忆、无偏好），需要通过系统调教使其成为懂用户需求的专属助手。这一过程无需编程知识，只需耐心和正确方法。

### **🔧 第一步：给龙虾一个清晰的身份**

OpenClaw的核心配置文件位于工作区（workspace），决定AI的"人格"和"边界"。

#### **工作区配置文件一览**
```
~/.openclaw/workspace/  
├── AGENTS.md        # 操作规则、优先级、行为准则  
├── SOUL.md          # 人格、语气、边界设定  
├── USER.md          # 用户画像与称呼偏好  
├── IDENTITY.md      # 名称、角色、能力定义  
├── TOOLS.md         # 工具使用说明  
├── MEMORY.md        # 长期记忆、知识沉淀  
└── memory/          # 记忆日志目录  
```
#### **关键配置文件详解**
1. **SOUL.md：定义AI的"灵魂"**  
   最重要的人格文件，决定AI说话风格、行为边界。  
   **示例结构**：
   - 角色定位（如"资深Python技术助手"）
   - 说话风格（如"简洁直接，代码优先"）
   - 能力边界（明确"能做"与"不能做"的范围）
   - 安全原则（如文件操作前必须确认路径）

2. **USER.md：让AI记住"你是谁"**  
   用户画像文件，包含：
   - 基本信息（姓名、角色、时区）
   - 技术栈（主语言、框架、工具）
   - 工作偏好（代码风格、提交规范）
   - 当前项目（开发/维护阶段）

3. **AGENTS.md：立下"行为准则"**  
   AI的操作手册，包含执行规则、工具使用规范、记忆管理和禁止行为。  
   **示例规则**：
   - 文件操作前确认路径存在
   - 删除前用`--dry-run`预览
   - API调用失败至少重试2次

### **🍎 第二步：喂给龙虾第一批"入职材料"**

有了身份后，需通过高质量知识投喂构建AI的专属知识库。

#### **喂食原则：质量 > 数量**

100个混乱文档的效果远不如10个精心整理的文档，AI只能从提供的材料中提取信息。

#### **喂食材料清单（优先级排序）**

| 材料类型 | 具体内容举例 | 重要性 |
| :--- | :--- | :--- |
| **公司基本信息** | 名称、主营业务、产品/服务、组织架构 | ★★★★★ |
| **OKR目标** | 个人/团队/协同团队OKR | ★★★★☆ |
| **常见问题FAQ** | 内部高频问题及标准答案 | ★★★★☆ |
| **业务术语表** | 行业专有名词、内部缩写（如"FP"="First Purchase"） | ★★★☆☆ |
| **工作流程SOP** | 会议督办、客户跟进、审批流程 | ★★★☆☆ |
| **边界规则** | 可直接回答/需转人工的问题界定 | ★★☆☆☆ |

#### **与其他龙虾交流：外部学习平台**

| 平台 | 特点 | 学习内容 |
| :--- | :--- | :--- |
| **moltBook** | OpenClaw官方知识库 | 官方文档、最佳实践、常见问题 |
| **Instreet** | 社区交流平台 | 实战案例、经验分享、问题解答 |
| **GitHub** | 开源项目 | 配置模板、插件开发、工具集 |
| **技术博客** | 专家经验分享 | 深度教程、架构设计、优化技巧 |

**学习建议**：设置定期学习任务，如"每周五下午浏览moltBook和Instreet，总结3个适合我的优化点"。

### **📚 第三步：让龙虾主动学习**

真正的区别在于**能否持续学习和成长**，需建立自我改进机制和学习循环。

#### **自我修养：建立"错题本"机制**

OpenClaw 2026.3.3版本引入self-improving-agent机制，通过`.learnings/ERRORS.md`记录错误：

**错误记录结构**：
- 错误描述（如"执行pip install时未确认依赖版本导致冲突"）
- 根因分析（缺少版本检查、未测试环境）
- 改进措施（检查requirements.txt、用虚拟环境测试）
- 升级到配置文件（更新AGENTS.md规则）

**自我改进时间表**：

| 时间 | 动作 |
| :--- | :--- |
| 24小时内 | 记录错误详情到ERRORS.md |
| 48小时内 | 实施改进措施并验证 |
| 每周 | 复习错误记录，提取通用规则 |
| 每月 | 将高频错误升级到核心配置文件 |

#### **专业知识学习：构建学习循环**

建立"学习-应用-反馈"闭环：
```
1.设定学习目标  
↓  
2.收集学习资料(文档、教程、案例)  
↓  
3.让AI阅读并总结  
↓  
4.实际应用(写代码、解决问题)  
↓  
5.反馈结果(成功/失败)  
↓  
6.记录经验到MEMORY.md  
↓  
7.回到步骤1(下一轮学习)
```
#### **主动学习习惯：3个建议**
1. **每日复盘**：总结当天问题、经验、改进点并更新到MEMORY.md  
2. **定期深度学习**：每周选主题（如"PostgreSQL索引优化"）进行专题学习  
3. **建立知识图谱**：连接零散知识（如FastAPI、PostgreSQL、Redis的关系）

### **🎯 第四步：让虾越来越聪明的小Tips**

养AI和养人一样，习惯决定成败，需养成三个关键习惯：

#### **习惯一：每日犯错 → 马上修**

**不要让错误积累**，错误记录标准做法：

| 时间 | 动作 |
| :--- | :--- |
| 24小时内 | 记录错误详情到ERRORS.md |
| 48小时内 | 实施改进措施并验证 |
| 广泛适用时 | 升级到核心配置文件 |

**触发记录的6种情况**：命令失败、用户纠正、请求不存在能力、API失败、知识过时、发现更好方法

#### **习惯二：定期喂新知识**

虾的知识是静态的，但业务是动态的，需定期更新：

| 频率 | 更新内容 | 示例 |
| :--- | :--- | :--- |
| 每周 | 新增常见问题和答案 | 新学技术技巧、踩坑经验 |
| 每月 | 产品/服务更新、流程变化 | 项目新功能、工作流程调整 |
| 每季度 | 全量知识库审查 | 清理旧项目资料、更新技术栈 |

**喂新知识方法**：收集→整理→放入memory目录→告知AI学习→让AI总结

#### **习惯三：备份是底线**

精心调教的配置是团队资产，需防止因负责人离职、账号迁移等导致丢失：

**备份方法**：
1. **Git版本控制**：初始化仓库并定期提交配置
2. **飞书/Notion文档备份**：复制关键配置并定期更新
3. **配置导出脚本**：编写Python脚本自动备份工作区

**备份频率**：重大修改后立即备份、每周自动备份、每月导出到文档

### **⚠️ 常见坑&解决方法**

| 问题 | 根因 | 解决方法 |
| :--- | :--- | :--- |
| 回答牛头不对马嘴 | 知识库无相关内容 | 补充FAQ，明确"不知道就说不知道" |
| 回答错误 | 知识过时或冲突 | 定期更新知识库，删除矛盾内容 |
| 团队没人用 | 不知虾能做什么 | 内部培训，演示3个实用场景 |
| 群里刷屏 | 触发规则太宽松 | 限定@或特定关键词时响应 |
| 交付空文档 | 无VBR原则（验证后报告） | 完成前必须从用户视角验证 |
| API第一次失败就放弃 | 缺乏坚持 | 至少尝试10次/替代方法再求助 |
| API Key明文存储 | 安全意识不足 | 用环境变量+.gitignore |
| 配置丢失 | 无备份机制 | 写入飞书文档，标准化交接 |
| 要求无效 | 反馈机制不完善 | 明确纠正错误/强化正确行为 |

### **💡 总结：虾不是工具，是伙伴**

OpenClaw的强大不在于初始能力，而在于"养"的过程。通过构建"数字分身"，它会：
- 记住工作习惯
- 理解思维方式
- 积累经验教训
- 代替重复劳动

**核心观点**："虾不是买来就会用的产品，是养出来的伙伴。"
