# 当升科技方案设计全流程 · 方法论萃取

> 来源：2026-05-18 全天方案设计交互过程
> 从 v3.0 问题诊断到 v5.0 定稿的完整设计路径

---

## 一、方案设计的核心原则（秦曼老师红线）

### 1.1 设计逻辑方向

**错误方向**：从现有老师和课程出发，往方案里塞内容
**正确方向**：理解客户业务和领导力挑战 → 明确培养对象 → 明确该层级需要具备的核心能力 → 围绕核心能力设计项目匹配对应课程

### 1.2 七部分方案架构

大咨询公司的项目建议书标准架构：
1. 对客户项目背景及需求的理解
2. 咨询公司整体解决方案
3. 培训目标及阶段设计
4. 课程整体设计
5. 实战成果输出
6. 项目运营管理与交付
7. 我们的团队

### 1.3 九大模块（秦曼确认）

管理者角色转换、绩效管理（目标设定/反馈辅导/评估激励）、高效会议、知人善任、授权、高效沟通、打造高绩效团队、影响力与团队协作、商业敏感度-财务基础

### 1.4 关键红线

- 项目建议书阶段不需要客户提供具体名单，只需明确新晋经理定义
- 不要强行加入国际化内容（人员不会全部马上派往海外）
- 必须明确项目定位（一年项目 vs 三年方案）
- 培训目标不能把课程安排错当目标
- 概念不能混淆：四大职责是工作内容，三大能力是条件，培养模块对应能力建设
- 当前阶段不需要匹配老师，先把框架逻辑理清楚
- 不确定的内容不要写，但可以委婉角度写
- 方案语言要正式书面化，不能口语化
- 不能有代码块格式，全部用表格

## 二、方案架构的核心设计决策

### 2.1 章节顺序的最终形态

一、客户背景与需求 → 二、整体解决方案（框架+逻辑链）→ 三、培训目标及阶段设计 → 四、课程整体设计 → 五、实战成果输出 → 六、项目成果沉淀与组织能力建设 → 七、项目运营管理与交付 → 八、摩根世纪核心优势 → 九、团队

逻辑：需求→方案→目标→课程→个人产出→组织沉淀→运营→优势→团队

### 2.2 章节归属的多次调整

| 调整项 | 初始位置 | 最终位置 | 原因 |
|--------|---------|---------|------|
| 培养目标 | 第二部分 | 第三部分 | 目标太细，和阶段设计单独成章 |
| 挑战-目标映射 | 第三部分开头 | 第二部分 2.3 | 这是逻辑论证，属于方案而非课程 |
| 核心优势 | 第三部分 | 第八部分 | 先讲完方案再讲"为什么是我们" |
| 新晋管理者发展路径 | 第五部分 | 删除 | 与第四节内容重叠 |
| 实战成果输出 | 第六部分子节 | 第五部分独立章节 | "产出什么"和"怎么管"分开 |
| 项目运营管理与交付 | 与实战成果混合 | 第七部分独立章节 | 同上 |

### 2.3 三阶段设计的关键转变

**初始设计**：管理半径递进——第一阶段管事、第二阶段带人、第三阶段建团队（不同阶段覆盖不同模块）
**问题**：新晋经理上任第一天就同时面对所有问题，不能"先学管事，三个月后再学带人"
**最终设计**：三阶段都覆盖全部九大模块，深度递进——概念建立→应用深化→整合输出

### 2.4 能力框架的表述策略

**初始问题**：自建5力模型，然后映射到四大职责/三大能力，多此一举
**秦曼反馈**："培养模块只需要对应能力建设即可，不需要拆分对应两类内容"
**最终方案**：摩根标准培养框架（三层递进）作为主体，当升四大职责/三大能力在"定制化适配"段落实对应——**框架是摩根的IP，当升在框架上定制**

### 2.5 培养目标与挑战的连贯性

**问题**：1.3节四类挑战、2.2节三层框架、3.1节培养目标，三者没有显性连接
**解决方案**：增加2.3节"从挑战到目标的逻辑链条"，用一张表将四类挑战归入三层框架，每层指向具体培养目标

## 三、从 DDI 萃取到当升应用的知识迁移

| DDI 设计概念 | 当升方案中的应用 |
|-------------|---------------|
| 递进模型（自我认知→业务管理→团队管理→人际管理） | 三层标准培养框架（管理能力→带队能力→经营能力） |
| 行为模式法（拆为可观测的关键行动） | 每个模块定义行为标准+验收方式 |
| 准备度概念（先测评再学习） | 第一阶段前置角色转换模块，所有模块的前置条件 |
| 最小可训单位（独立可组合课程） | 九大模块各自独立，企业可按需组合 |
| 认知缺口驱动（诊断→学习→模拟→考核） | 每阶段课前诊断→集中学习→实战产出→阶段考核 |
| 改变可视化（前后测对比） | 每个模块有验收方式，第三阶段面向管理层述职 |
| 痛点即入口（先用痛点抓目标） | 1.3节四类挑战→2.2节框架回应→2.3节映射到目标 |

## 四、方案语言规范

### 要避免的表述
- 口语化："怎么定、怎么开、怎么谈" → 改为正式表述
- 代码块：``` → 改为引用块或表格
- "便同时""即需" → "同时""需"
- 未经客户文件确认的概括性描述

### 要采用的表述
- 表格优于代码块
- 客户原文优于自行概括
- "基于摩根对同类企业的长期观察"优于"我们理解"
- 结构化表格优于长段落叙事
