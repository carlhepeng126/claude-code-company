# AI业务专家验证：让Claude按我的方法论诊断客户需求

> **目标**：不建 agent，用现有工具（Claude Code + Obsidian 知识库）验证"我的方法论能否让 AI 做出靠谱判断"。跑通一个场景，10 次真实测试，再决定是否 agent 化。

## 为什么是这个顺序

方法论结构化 → AI 验证 → agent 化，这个顺序不能跳。Agent 只是放大器——如果你还没验证 AI 能不能按你的框架出活，建 agent 就是花钱放大不确定的东西。

## 选哪个场景

**企业出海培训需求诊断**——客户找上门，你需要在 30 分钟内判断：
- 客户的真实问题是什么（不是他说了什么）
- 问题属于哪个类型（能力缺口型 / 业务倒逼型 / 政策驱动型）
- 应该推什么方案（单次培训 / 系统项目 / 持续服务）
- 报价区间和师资配置

这是你最高频、最成熟的判断场景。先把这个跑通。

**要新建的卡：**

> **B024-企业出海培训需求诊断决策流**

内容结构建议：
```
1. 需求分类三问（3个问题锁定客户类型）
2. 客户画像速写表（行业×出海阶段×人员规模×紧迫度）
3. 短板快速定位矩阵（语言/专业/管理/文化 四象限）
4. 方案类型决策树（单次培训/系列课程/系统项目/持续服务）
5. 师资配置速查表（四角色×预算档位）
6. 报价参考区间（按方案类型×天数×师资数）
```

### 二、黄金案例（最少3个）

**素材来源：** `LLM Wiki知识应用/raw/培训案例/`（70+ 央企培训原始报道）

从已完成项目中选 3 个，完整记录：

```
输入：客户初始需求描述（原话）
↓
诊断过程：你问了什么、发现了什么
↓
输出：最终方案（类型/天数/师资/报价/理由）
```

建议选用（覆盖三种方案类型）：

| 类型 | 推荐案例 | 素材文件 |
|------|----------|----------|
| **体系型** | 中原对外工程 | 已有完整萃取 [[B016]] |
| **持续服务型** | 当升科技 | 已有完整萃取 [[B017]] |
| **单次培训型** | 待定——从培训案例库中选一个 | 见下方候选 |

**单次培训型候选（从案例库初筛）：**
- 中交集团第一期海外市场开发人才培训班（10天/99人/五大模块）— `[202309191627]`
- 中核二四国际化工程项目英语人才脱产培训 — `[202211111847]`
- 中国能建华北院高级英语培训班 — `[202305202145]`

> 选一个你亲身参与过、记得诊断决策过程的。

**黄金案例提取模板：**
1. 从 `raw/培训案例/` 复制原文到案例文档
2. 补充"诊断过程"部分（这是原文没有的——你当时怎么判断的）
3. 补充"方案决策理由"（为什么是这个方案，不是别的）

### 三、输出模板（1页纸）

定义 AI 诊断结果的标准格式，让每次输出可直接对比：

```
## 客户诊断卡
- 客户类型：[ ]
- 真实需求：[ ] （客户说的 vs 实际需要的）
- 短板定位：[ ] （四象限标注）
- 方案类型：[ ]
- 师资配置：[ ]
- 报价区间：[ ]
- 风险提示：[ ]
- 下一步动作：[ ]
```

### 四、10个测试用例

从 `raw/培训案例/` 的 70+ 案例中选 10 个，匿名化为"客户初始描述"。

**覆盖维度：**
- 央企 vs 地方国企 vs 民企
- 工程类（中交/中核/中电建）vs 能源类（中海油/中能建/中铝）vs 科技制造类（华为/比亚迪/北汽）
- 英语基础培训 vs 商务英语 vs 海外市场开发 vs 国际化领军人才
- 单次项目 vs 系列培训 vs 体系化项目

**提取方法：**
1. 从案例原文中只提取 **"客户是谁 + 他们说要什么"**（1-2 句话）
2. 不包含你的诊断和最终方案（那是验证 AI 用的）
3. 另建一个"标准答案"文件记录你实际的诊断结果，测试时对照

## 执行节奏

| 周次  | 做什么                           | 产出             |
| --- | ----------------------------- | -------------- |
| 第1周 | 写 B024 决策流 + 整理3个黄金案例         | 1张卡 + 3个案例文档   |
| 第2周 | 写输出模板 + 整理10个测试用例             | 1页模板 + 10条测试输入 |
| 第3周 | 用 Claude Code 跑 10 个测试，记录对比结果 | 10组对比记录        |
| 第4周 | 复盘：准确率、失败模式、框架修正              | 验证报告 + 决策      |

## 成功标准

- 10 个案例中 ≥8 个 AI 诊断与你的判断**方案类型一致**
- 短板定位**象限正确** ≥8 个
- 没有"完全跑偏"的诊断（比如该推持续服务的推了单次培训）

## 如果验证通过

1. 把决策框架打磨成可复用的 `/diagnose-client` Skill
2. 在真实新客户上试用（不替代你的判断，只做辅助）
3. 积累 20+ 案例后，再评估是否 agent 化

## 如果验证不通过

复盘失败模式——是框架没写清楚，还是这个判断本身依赖太多不可编码的直觉？前一种就迭代框架，后一种就接受边界、缩小 agent 范围。
