# 当升科技项目复盘：方案设计与课程交付标准

## 一、项目全景

| 维度 | 咨询案（轮岗与激励） | 培训案（新晋经理） |
|------|-------------------|-------------------|
| 客户触发场景 | 业务高速扩张，人才跨部门/跨地域流动无制度支撑 | 昆明8.30投产紧急需求，48名新晋管理者能力缺口大 |
| 方案形态 | 制度设计咨询（8-10周） | 12个月领导力培养项目 |
| 模型 | 4D: Diagnose → Design → Deliver → Deploy | 70-20-10 + 三阶段递进 |
| 与客户战略咬合 | 四大职责三大能力 → 轮岗制度承载 | 四大职责三大能力 → 胜任度模型 → 课程设计 |
| 客户状态 | 三家比价 | 三家比价 |

## 二、方案设计过程复盘

### 2.1 需求诊断阶段的关键动作

| 动作 | 产出 | 对后续设计的影响 |
|------|------|----------------|
| 从客户会议中提取董事长"四大职责三大能力"框架 | 战略锚点映射表 | 方案不再用通用管理语言，而是用客户自己的语言体系——降低理解成本 |
| 识别"芬兰外派"只是表象，真实问题是"全公司跨部门轮岗" | 轮岗范围从云南→全公司 | 方案骨架从"外派制度"升级为"人才流动体系" |
| 秦总反馈：现有方案"大而全、模块逻辑混乱" | 课程结构从"用天数框内容"改为"从需求倒推课程" | 去掉涉外谈判/经营沙盘（后保留），绩效管理从Phase 2前置到Phase 1 |
| 发现客户芬兰项目不仅是"培训外派人员" | 1.3节从抽象优势变为具体证据表 | 常州+芬兰实地+双向融合→课程设计有真实场景做素材 |

### 2.2 方案骨架设计原则

**原则1：先定义标准，再设计内容**

```
胜任度模型（13项能力×行为标准）
     ↓
三阶段框架（按"基础→应用→进阶"分配能力）
     ↓
每阶段课程表（每个模块反向对应一项胜任力）
```

任何课程方案，如果连"学完应该变成什么样"都没说清楚，课程表再好看也没用。

**原则2：用客户的语言写方案**

| 通用培训语言 | 当升方案中的对应 |
|------------|----------------|
| "干部管理能力提升" | "四大职责：建组织、搭流程、带队伍、做业绩" |
| "战略对齐" | "攻山头、打胜仗、练队伍" |
| "人才体系建设" | "飞机在飞速中换引擎" |

客户董事长在会议上反复使用的词汇，直接嵌入方案框架——不是迎合，是降低方案的认知成本。

**原则3：报价之前先有产出物清单**

Phase 1结束后HR能拿给董事长看的东西（6项速赢成果），在方案设计阶段就确定。HR汇报时不需要"解释方案"，只需要"展示结果"。

### 2.3 课程设计的关键决策点

| 决策点 | 最初方案 | 调整后 | 调整依据 |
|--------|---------|--------|---------|
| 国际化经营（刘德冰） | Phase 1 Day1 | 转为项目总顾问 | 不适合一线新晋管理者 |
| 涉外谈判（Ben Phone） | Phase 1 Day7 | 删除 | 一线经理不需要谈判技能 |
| 跨文化沟通 | 无 | Phase 3 新增 | 芬兰项目实际需要，但不是谈判 |
| 绩效管理 | Phase 2 | Phase 1 Day6 | 秦总反馈：核心模块不应放第二阶段 |
| 企业经营沙盘 | 无→删除→恢复 | Phase 3 Day3 | 客户当天反馈好，但定位为"全局视角"而非高管沙盘 |
| Phase 3 天数 | 4天→5天（增加内容后） | 5天 | 净增1天：加团队/影响/跨文化(2天)，压缩领导力/总复盘(1天) |

## 三、可复用的课程设计标准

### 3.1 方案结构标准

面向央企/国企客户的方案，按以下结构编写：

```
1. 项目背景与需求理解（证明你听懂了他的问题）
2. 培养对象（明确边界——谁在、谁不在）
3. 培训目标（一句话：从A变成B）
4. 岗位能力胜任度模型（先定义"好"的标准）
5. 问题导向表（从痛点→设计→验收的完整链）
6. 时间与分班安排
7. 能力培养框架（三阶段递进逻辑）
8. 三阶段课程设计（每阶段：目标→课程表→产出）
9. 师资配置（外部+内部，明确分工边界）
10. 培训方式（70-20-10锚定行业标准）
11. 项目运营与品牌化
12. 线上线下融合机制
13. 培训闭环与业务产出（学→用→查→验的完整链路）
14. 项目报价（分阶段、分项目、分批次）
```

### 3.2 课程模块筛选标准

判断一个模块是否应该放入方案，用以下三个问题：

1. **层级匹配吗？** 这个模块适合"一线新晋管理者"还是"中高层"？——角色认知放Phase 1 ✓，经营沙盘放Phase 3（精简版）✓，涉外谈判——不放
2. **有业务场景吗？** 这个模块能否用客户自己的案例做素材？——能→放，不能→换通用案例但标注风险
3. **有产出物吗？** 学完这个模块，学员手上多了一个什么可以直接用的东西？——有→放，没有→重新设计模块内容

### 3.3 三阶段能力分配规则

```
Phase 1（破局）：基础生存技能
   └─ 角色转型 + 带团队基本功 + 绩效管理入门
   └─ 标准：学完就能在日常管理中马上用到

Phase 2（攻坚）：实战应用能力
   └─ 深化 Phase 1 的技能 + 跨部门协同 + 问题分析
   └─ 标准：能独立处理真实管理场景的复杂问题

Phase 3（收官）：整合进阶素养
   └─ 复盘前两阶段 + 团队领导力 + 全局视角
   └─ 标准：能向高管做述职级汇报，能带新人
```

核心规则：**不是按课程难度分配，是按学员的成长节奏分配。** Phase 1 解决"会不会"的问题，Phase 2 解决"好不好"的问题，Phase 3 解决"能不能教别人"的问题。

### 3.4 实战嵌入标准

每门课程在方案中需明确回答三个问题：

| 问题 | 示例 |
|------|------|
| 练什么？ | 课堂案例：当升昆明投产的真实跨部门协调场景 |
| 产出什么？ | 课后任务：每人完成一份部门目标分解表 |
| 谁检验？ | 验收人：直线上级签字确认 |

没有这三个答案的模块，就是"讲理论的模块"——在一份实战导向的方案里不应该出现。

## 四、咨询案设计标准

### 4.1 4D 模型适用场景

```
Diagnose（诊断）→ Design（设计）→ Deliver（交付）→ Deploy（落地）

适合：客户需要的不只是一份报告，而是能落地执行的制度方案
不适合：客户只要一份诊断报告或对标分析
```

### 4.2 方案中必须出现的三个要素

1. **客户语言映射**：把客户的战略术语翻译成方案中的设计逻辑。不是拍马屁，是降低认知摩擦。

2. **首轮试点**：制度类方案最容易死的地方是"完美定稿后无人执行"。方案中必须内置试点机制——范围小、成本低、输得起、能迭代。

3. **成功标准**：不写"制度发布"，写"有人动起来了、动了之后更好、不需要我们一直在场"。

### 4.3 报价原则

- 阶段报价优于整体报价（让客户先验证第一阶段质量）
- 参考课时优于固定课时（范围扩大时不会被动）
- 陪跑选项独立于项目报价（不给客户"必须一次性承诺"的压力）

## 五、客户沟通中的避坑指南

| 坑 | 表现 | 怎么避 |
|----|------|--------|
| **需求表里不一** | 客户说"办一期英语培训"，实际要的是"跨文化融合+SOP沟通+工具赋能" | 不问"你要什么课"，问"你的人在什么场景下觉得最吃力" |
| **方案大而全** | 把情境领导力、涉外谈判、经营沙盘全塞进一线管理者课程 | 先确定目标人群的管理层级，再筛选模块。一线≠中高层 |
| **用天数框内容** | "每个阶段7天，往里面填课程" | 从需求倒推内容，从内容倒推时长。客户不会因为你天数多就选你 |
| **方案的"秀感"** | 拿通用方法论包装成"独门秘籍"、攻击竞品、过度修饰 | 陈述事实，呈现设计逻辑，不自我标榜。务实客户最反感"会说话但不会干活" |
| **忽略内部HR的处境** | 方案只考虑向董事长汇报，没考虑HR怎么向领导交差 | 每个阶段结束时给HR一份可直接提交的"速赢成果清单" |
| **培训与业务两张皮** | 课程内容通用化，和客户当前业务挑战无关 | 课题选题绑定客户实际业务里程碑（如昆明8.30投产） |

## 六、沉淀为后续项目的检查清单

### 6.1 方案启动前

- [ ] 客户有没有成文的战略/领导力框架？有→直接引用；没有→帮客户提炼
- [ ] 客户在比价吗？是→方案中必须有一句话能成为董事长的拍板理由
- [ ] 目标人群的管理层级确认了吗？一线≠中高层≠高管
- [ ] 客户有没有已有的培训/咨询经历？有哪些坑？
- [ ] 竞品可能在哪个点上攻击我们？

### 6.2 方案设计中期

- [ ] 胜任标准在课程设计之前就定义清楚了吗？
- [ ] 每个模块都能回答"练什么、产出什么、谁检验"吗？
- [ ] 分班策略和报价一致吗？（几班次×几天=总报价）
- [ ] 有没有不适合目标人群层级的模块？
- [ ] 方案中所有天数加总一致吗？（课程表、概览表、时间表、70-20-10表）
- [ ] 有没有"待定"的讲师或内容？

### 6.3 方案交付前

- [ ] 方案中有没有"秀"的表达？（"独一无二""无可复制""业界领先"→删）
- [ ] 客户的语言出现在方案中了吗？（不是迎合，是降低认知成本）
- [ ] HR拿着方案能直接向领导汇报吗？（需要速赢成果清单）
- [ ] 方案的结构参考了大咨询公司标准吗？（背景理解→框架→设计→执行→团队→报价）
- [ ] 报价逻辑清晰吗？（不是什么钱都算清楚，而是客户能理解"为什么是这个价"）

## 七、关联核心卡与知识库

本次当升项目的设计过程中，以下核心卡（B卡）提供了关键判断框架，同时本次实践也反向验证和丰富了这些卡片：

| 核心卡 | 在本次项目中的应用 | 本次实践的反向补充 |
|--------|------------------|------------------|
| [[B017-从单次培训到持续服务-当升项目萃取]] | 芬兰单次培训→轮岗咨询+新晋经理培训，验证了"交付终点是下一个需求起点"的增长逻辑 | 补充了"两个项目如何联动"的具体做法（制度+能力双轮驱动） |
| [[B010-当升项目落地断点管理]] | 方案中内置首轮试点、成功标准、速赢成果清单——都是断点管理思维的具体落地 | 验证了"需求衰减、协同摩擦、效果滞后"三个断点的识别框架 |
| [[B024-企业出海培训需求诊断决策流]] | 芬兰项目经验反向导入方案设计（1.3节证据表），验证了诊断框架的"场景挖掘优于需求确认"原则 | 新增了一个诊断动作：从客户董事长公开讲话中提取战略语言 |

## 八、设计过程时间线

本次两个方案的设计周期为 2026年5月12日-15日，共4天：

| 日期 | 轮岗咨询方案 | 新晋经理培训方案 |
|------|------------|----------------|
| 5/12 | 完成初稿（基于夏楠、袁淑萍大纲） | 沿用旧版框架 |
| 5/14 | 7人会议（袁淑萍参会），客户反馈：轮岗范围扩大、方案可落地性需强化 | 5人会议（秦总主导），秦总提供苏伊士/施耐德/诺华标杆案例，指出方案四大问题 |
| 5/15 上午 | 内部沟通：报价策略、项目周期调整、人员分层宣贯 | 秦总输出方案框架大纲，摩根团队同步调整 |
| 5/15 下午-晚间 | 多轮迭代：扩大轮岗类型、新增访谈方法论、加入首轮试点、调整报价 | 多轮迭代：重构课程表、新增胜任度模型、加入70-20-10、补回企业沙盘、新增跨文化沟通 |
| 最终交付 | 14节结构，8-10周，4D模型 | 14节结构，12个月，三阶段递进 |

关键观察：两个方案都在4天内从"可用"迭代到"可提交"，核心原因是——
- 有B017的增长逻辑做方向锚定（不需要从头想策略）
- 有B010的断点管理做风险预判（首轮试点、成功标准、速赢清单都是提前内置的）
- 客户反馈密集且具体（三次会议、两个纪要、秦总现场诊断）
