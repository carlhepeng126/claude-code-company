同一个词，大家默认的“世界”不一样  
最开始出问题的是一个很普通的词：approval。  我当时理解的是：设计做完 → 提交 → 对方确认 → ok。但实际项目里，approval有三层意思：

  * 有时候是内部工程师点头 

  * 有时候是咨询公司正式盖章 

  * 有时候是政府部门审批通过 

我们在等“approval”，对方觉得已经给过了；对方在催“approval”，我们以为还没到那一步。  大家用的是同一个词，但脑子里的流程完全不同。
后来慢慢发现，这不是个例。

  * submission，我们以为“发过去就算交了”；对方那边是“系统登记 + 编号 + 回复确认 + 审查周期启动”。 

  * coordination，我们觉得“拉个会对一下模型”；他们默认是“会议纪要 + action list + 责任人 + follow-up”。 

  * review，我们习惯内部自审；他们很多时候指的是正式审查流程的一部分。 

  * detailed design，更离谱。国内团队一开始听到 detailed，容易自动联想到“施工图深度”甚至“深化图深度”。但在很多英式或国际项目体系里，它更多是 Technical Design 的意思，是把设计、规范、材料、系统协调到可以审查、招标或继续形成施工信息的程度。它不一定等于 IFC drawing，更不等于 contractor shop drawing。这个边界如果不提前说清楚，后面就很容易出现：业主觉得你还没做到 detailed，你觉得对方已经在要求施工深化。 

这些词没有一个是难词，但几乎每个都踩过坑。

问题不在英语，在“系统”  
我后来意识到一个很不舒服的点：我们不是在用英语沟通，而是在用中文工程逻辑，说英文单词。  对方也一样——用他们自己的体系，在说同样的词。
表面上是语言统一了，实际上是系统冲突。  比如我们习惯的是一种“线性流程”：  设计 → 提交 → 审批 → 出图  但很多国际项目更像是一个循环系统：
submission → review → comment → resubmission → re-review
你如果用线性的理解去做循环系统的事情，就会一直觉得“为什么老是返工”。  再比如责任边界。
我们默认coordination是“大家一起解决问题”；但很多项目里，它其实是一个责任分配机制：

  * 谁提问题？ 

  * 谁关闭？ 

  * 谁承担后果？ 

如果你没意识到这一点，就会不自觉地在帮别人兜底。

一开始我以为是沟通问题，后来发现是结构问题  
刚开始我也试过“加强沟通”：  多开会、多确认、多解释。  效果有限。  因为问题不在“有没有说清楚”，而在大家默认的结构不同。  有一次很典型。
对方说：“Please submit the drawings for review.”
我们理解是：把图纸发过去。他们理解是：正式提交一个版本，进入审查流程。
结果我们发了邮件，他们那边没有登记；他们等系统里的submission，我们觉得已经交了。  中间空转了差不多一周。  没有人犯错，但事情就是卡住了。
那一刻我才有点明白：沟通成本其实不是信息传递成本，而是“系统对齐成本”。

后来我开始改做一件很笨的事  
不再假设大家理解一样。  每次碰到关键节点，我都会多问一句：
“这个approval，是哪一级的？”“这个submission，需要走系统吗？”“这个review，是内部还是正式审查？”  有时候甚至会直接写下来：

  * what is included 

  * who approves 

  * what is the output 

刚开始觉得有点多余。后来发现，这一步能省掉后面80%的反复。  还有一个变化是，我开始少用词，多描述动作。
不说“我们完成了coordination”，而是说“我们已经和结构确认了梁高，更新了模型，并关闭了相关clash”。  词是模糊的，动作是具体的。

有些成本，其实是看不见的  
现在回头看，很多项目拖慢的地方，并不是技术难度。  而是这些“看不见的错位”：

  * 同一个词，不同的边界 

  * 同一个流程，不同的起点 

  * 同一个结果，不同的判断标准 

你如果只盯着图纸、模型、进度表，很容易忽略这些。  但真正消耗精力的，反而是这些地方。

最后一个小判断  
后来我有个很粗糙但挺有用的判断标准：  当一个词开始反复被提起，但事情没有推进，那大概率不是执行问题，是大家在不同系统里说话。
这个时候，继续解释意义不大。得先把“系统”对齐。  不然你会一直觉得，对方在拖；对方也会觉得，你没按规则来。
其实大家都在按自己的逻辑做事，只是用的不是同一套规则。
