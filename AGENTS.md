### 路由逻辑

| 场景 | 走编排器 | 直接走子Agent |
|------|:--:|:--:|
| 用户只说英语培训/管理咨询 | ✓ | - |
| 用户描述同时涉及两个域（如"海外项目经理不会带团队"） | ✓ | - |
| 用户说"完整方案""全案""双轨""语言+管理" | ✓ | - |
| 用户查询央企培训机会/竞品/行业（"看看XX有没有机会"） | ✓（优先走intel-expert） | - |
| 用户明确用 `/diagnose-client` 或 `/consult-client` | - | ✓ |
| 用户明确用 `/intel-expert` | - | ✓ |

编排器检测到情报查询信号时，优先路由到 intel-expert 出情报卡，再视情况继续路由到 diagnose/consult。原则：宁可多走编排器。

## LLM Wiki 运行机制

Wiki 操作分为四层（详见 [SCHEMA.md](LLM%20Wiki知识应用/SCHEMA.md)）：

| 层级 | 触发条件 | 目标 |
|------|---------|------|
| **Ingest（摄入）** | 有新资料到达 | 提取价值 → 更新或创建 Wiki 页面 → 维护索引和链接 |
| **Query（查询）** | 跨页面综合问题 | 检索多个页面 → 综合回答（带引用）→ 标注矛盾 |
| **Archive（归档）** | Query 产生有价值结论 | 存回知识库 → 双向关联 → 越用越丰富 |
| **Lint（健康度审计）** | 每次 Ingest 后 / 每月 | 检查孤立页面、过时内容、悬空链接、index 同步 |

**默认工作流：all output → wiki first。** 所有为 Carl 生成的内容（报告、分析、方案、总结）必须摄入到 Wiki，要么新建页面，要么更新现有页面。Wiki 是唯一真相源，不入库不算完成。例外：一次性回复、快速确认、临时数据。

## ACT ↔ LLM Wiki 互通规则

三个方向的互通（详见 SCHEMA.md）：

1. **ACT 项目 → Wiki**（项目启动时）：ACT 项目卡片必须包含 `📎 知识库关联` 区块，关联 2-3 篇最有价值的 Wiki 素材
2. **Wiki → ACT**（反向标注）：entities/concepts 页面的 frontmatter 中 `related_projects` 字段标注哪些项目用了该知识
3. **项目交付 → 知识沉淀**（项目完成后）：从 ACT 萃取出 entity 存入 LLM Wiki，格式为 `cases-<项目类型>.md`

## 我的工作方式（AI 请遵守）
1. **目标先行** — 任何问题按"目标→策略→流程→分工"顺序，不跳过
2. **资源前置** — 做事前先看知识库有什么资源可用，不凭空起方案
3. **轻试行，再复盘** — 不追求完美，先跑通再沉淀
4. **不杜撰** — 没有真实来源的数据/案例/人名，绝不编造。不确定就说"不确定，需要查证"
5. **不空话** — 每个建议必须指向可执行的动作或判断标准
6. **日归档** — 每次对话开始时，检查昨天的日日志（`5-Time/5-4-Daily-日计划/`）和 Action 文件夹，将已完成事项移入归档

## B卡（核心卡）体系
B卡（B000-Bxxx）是核心思维框架，存储在 `ACT笔记-深度思考/4-Card/4-3-MainCard-核心卡/`。每张卡对应一个方法论或经验萃取，如 B008 客户需求场景化拆解、B016 央企出海培训方案设计框架。在做方案设计和客户分析时优先查阅对应的 B 卡。

## 输出偏好
- 中文为主，术语保留英文缩写（LDP/SOP/KPI）
- 复杂逻辑优先用表格 + 结构化清单
- 做真正能解决客户问题的方案，凡确定事项一定能落地
- 反馈时使用核心卡（B编号）作为框架参考
