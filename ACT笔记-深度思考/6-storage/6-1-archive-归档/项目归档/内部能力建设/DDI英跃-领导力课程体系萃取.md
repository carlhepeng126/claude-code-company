# DDI 英跃领导力课程体系 · 设计逻辑深度萃取

> 萃取来源：DDI 英跃官网（2026年5月）
> 聚焦：体系模型、痛点-方案逻辑、课程-目标映射、核心设计概念

---

## 一、DDI 的底层体系模型

### 1.1 核心命题

DDI 整个体系回答的是一个根本问题：

> **"一个基中层管理者，从'自己能做事'到'带团队拿结果'，需要跨越什么？"**

他们的答案是三层跨越：

```
从"自己做" → 到"带人做" → 到"跨边界推动做"
   (业务管理)      (团队管理)        (人际管理)
```

这不是三个平行的"课程分类"，而是一个**管理者成熟度的递进模型**。

### 1.2 体系模型图

```
                    ┌──────────────────────────┐
                    │  人际管理：跨边界影响力    │
                    │  "事情不由你管，但你要推得动" │
                    │  协作 · 影响力 · 冲突解决   │
                    ├──────────────────────────┤
                    │  团队管理：通过他人拿结果   │
                    │  "你不是做事的人，你是让人做事的人" │
                    │  选才·辅导·授权·建队·变革   │
                    │  留人·虚拟团队·育才         │
                    ├──────────────────────────┤
                    │  业务管理：自己先把事做成   │
                    │  "管人之前，先管好自己的事"  │
                    │  计划·决策·绩效·执行·创新   │
                    ├──────────────────────────┤
                    │  自我认知：知道你是谁       │
                    │  "不了解自己，管不了别人"    │
                    │  DNA测评 · 启程 · 精要      │
                    └──────────────────────────┘
```

**层级之间的递进逻辑**：

| 跨越 | 核心转变 | 对应课程域 |
|------|---------|-----------|
| 第0层→第1层 | 从"执行力"到"计划力"——不仅仅完成被分配的任务，而是能自己规划、决策、对结果负责 | 业务管理 |
| 第1层→第2层 | 从"我做好"到"他们做好"——放下"自己上"的冲动，学会选对人、教会人、放手让人干 | 团队管理 |
| 第2层→第3层 | 从"管自己人"到"影响不是自己人的人"——没有汇报关系，也要推动协作、化解冲突、赢得承诺 | 人际管理 |

**这个模型的核心洞察**：每一层跨越都不是知识问题，而是**身份认同的转变**。DDI 的课程设计本质上是在帮助管理者完成这些身份转变。

### 1.3 为什么必须先做"自我认知"？

DDI 把 DNA 测评、启程、精要三门准备课放在最底层，这是一个**有意图的设计选择**，不是简单的"入门课"。

背后的设计逻辑：

> 管理者的第一个问题不是"我不会管"，而是"我不知道自己适不适合管、该用什么方式管"。

- **DNA** 回答："你是一个什么样的管理者？"（大五人格→管理风格→盲区识别）
- **启程** 回答："管理者到底应该做什么？"（从专家思维到领导者思维的转换）
- **精要** 回答："管理者的基本功是什么？"（角色认知、关系处理、自我修养）

三门课形成一个**自我认知三角**：人格认知 + 角色认知 + 能力认知。只有当这三个认知都建立起来，后续的技能学习才有"锚点"——学员才知道"这门课对我意味着什么"。

## 二、痛点-方案映射逻辑

### 2.1 DDI 如何定义"痛点"

DDI 的课程设计有一个清晰的问题识别框架。他们定义的"痛点"不是笼统的"领导力不足"，而是**具体到行为层面的管理困境**。

分析他们每门课程的痛点表述，可以归纳出四类核心痛点：

#### 类型一：角色转型痛（"我不知道我现在该做什么"）

| 典型痛点 | 根因 | 对应课程 |
|---------|------|---------|
| "从聚焦于自己获得认可，不知道如何成就他人" | 身份认同未转换 | 领导力启程 |
| "授权还是不授权？总是不放心" | 放手的恐惧 | 授权 |
| "每天忙于处理可交给他人完成的任务" | 不会做"教练"而非"球员" | 辅导 |

**设计逻辑**：这类痛点不需要技能培训，需要的是**认知重构**。所以启程/授权/辅导课程的重点是先改变"管理者如何看待自己的角色"，再教方法。

#### 类型二：方法缺失痛（"我知道该做，但我不会做"）

| 典型痛点 | 根因 | 对应课程 |
|---------|------|---------|
| "面试中应该如何提问？应聘者给的信息可信吗？" | 缺乏结构化面试方法 | 目标选才 |
| "如何从无到有制定行动计划？" | 缺乏系统规划方法 | 计划与组织 |
| "花了很长时间辅导下属，但还是没啥改善" | 缺乏有效辅导方法 | 辅导 |

**设计逻辑**：这类痛点是可以直接给工具的。DDI 的做法是**把方法拆成步骤**（如"沟通五步法""决策五步骤"），每一步对应一个可操作的动作。

#### 类型三：系统缺失痛（"我一个人会做，但团队不会"）

| 典型痛点 | 根因 | 对应课程 |
|---------|------|---------|
| "组织战略目标与下属个人绩效目标相差甚远" | 缺乏绩效分解系统 | 达至最佳绩效 |
| "团队缺乏清晰的工作目标和衡量进展的措施" | 缺乏执行追踪系统 | 执行力 |
| "成员能力不错，合作起来不如人意" | 缺乏团队运行机制 | 打造高效团队 |

**设计逻辑**：这类痛点需要的不是个人技巧，而是**建立一套团队运行的系统**。DDI 的对应课程教的不是"你应该怎么做"，而是"你应该建立什么样的机制"——流程、章程、指标、问责制。

#### 类型四：关系困局痛（"我自己可以，但别人不配合"）

| 典型痛点 | 根因 | 对应课程 |
|---------|------|---------|
| "配合的部门总说没有资源，导致计划无法执行" | 缺乏非职权影响力 | 影响力 |
| "组织内各职能壁垒分明，大家都各自为营" | 缺乏跨部门协作能力 | 协作 |
| "下属闹矛盾，彼此难合作，影响团队士气" | 缺乏冲突处理能力 | 解决冲突 |

**设计逻辑**：这类痛点涉及的是**横向关系管理**——如何在没有正式权力的情况下推动事情。DDI 把这放在体系的最高层，隐含的判断是：**只有先管好自己的事、带好自己的团队，才有资格和能力去解决跨边界的问题**。

### 2.2 痛点→方案→目标的映射逻辑

DDI 的每门课遵循一个统一的设计公式：

```
一个具体痛点 → 一个可识别的原因 → 一组可操作的行为 → 一个可衡量的行为改变
```

以"辅导"课程为例拆解这个公式：

| 层级 | 内容 |
|------|------|
| 痛点 | "花了很长时间辅导下属，但还是没啥改善" |
| 原因 | 管理者分不清"问题是什么"和"问题的症状"，用同一种方式对待所有辅导场景 |
| 方案 | 区分三种辅导类型→识别根因→轻度辅导用"反馈三原则"→深度辅导用"沟通五步法" |
| 目标 | 管理者能够诊断下属的真实问题，选择合适的辅导方式，开展有效的辅导对话 |
| 衡量 | 辅导前后下属的行为变化可被观察到 |

**这个公式的关键**：不是"学完课程→能力提升"，而是"识别问题→选择工具→改变行为→看到结果"。DDI 卖的本质上是一个**行为改变的因果链**，课程只是中间的载体。

## 三、课程-目标映射关系

### 3.1 DDI 定义的目标体系

DDI 的16门核心课程，每门课背后有一个**行为层面的明确目标**，不是模糊的"提升XX能力"。

| 课程 | 行为目标（学员学完后能够……） | 目标的组织价值 |
|------|---------------------------|-------------|
| 计划与组织 | 制定包含任务分解、优先级、资源配置的行动计划，并建立追踪指标 | 减少返工和资源浪费 |
| 决策 | 运用五步骤流程做出业务决策，辨识并避开四种决策盲点 | 提升决策质量和速度 |
| 达至最佳绩效 | 与下属就SMART目标、能力要求、发展计划达成共识 | 组织战略分解到个人 |
| 执行力 | 运用"聚焦-衡量-分工"三要领确保战略落地 | 战略不悬空 |
| 推动创新 | 作为"创新孵化者"，从心脑眼手脚五维推动团队创新 | 组织保持创新活力 |
| 目标选才 | 运用STAR方法收集有效行为证据，评估动力适配性 | 选对人，降低流失 |
| 辅导 | 识别真正需要辅导的问题，选择合适的辅导方式，开展有效辅导对话 | 下属能力持续提升 |
| 授权 | 根据任务类型和人员匹配度进行有效授权，并跟进确保完成 | 管理者时间释放到更重要的事 |
| 打造高效团队 | 诊断团队在五要素上的健康度，运用六项关键行为改善团队效能 | 团队1+1>2 |
| 驱动变革 | 运用三大加速器化解变革抵触，推动团队从抗拒到拥抱 | 变革成功落地 |
| 激励与保留 | 运用三大驱动因子激励下属，识别离职风险信号并干预 | 降低核心人才流失 |
| 领导虚拟团队 | 为虚拟团队建立协作环境、沟通机制和成果追踪体系 | 远程团队效能不衰减 |
| 培养人才 | 运用3A流程为下属制定并实施发展计划 | 人才梯队持续造血 |
| 协作 | 运用伙伴关系五要素建立并维持跨部门协作关系 | 打破部门墙 |
| 影响力 | 综合运用策略、包装和承诺三要素赢得利益相关者支持 | 非职权推动事情发生 |
| 解决冲突 | 运用五种策略模型和六项关键行动将冲突转化为共识 | 化内耗为动力 |

### 3.2 目标映射的三个层次

DDI 的课程目标不是单层的，而是三层映射：

```
组织价值层    "这门课解决什么业务问题？"
    ↑
行为改变层    "学员学完后能做什么不一样的事？"
    ↑
技能习得层    "学员需要掌握什么知识和技巧？"
```

以"目标选才"为例：

| 层次 | 内容 |
|------|------|
| 技能习得 | 掌握STAR提问法、动力适配性评估、面试分工表的使用 |
| 行为改变 | 面试时不再凭感觉，而是系统收集行为证据；关注应聘者的动机匹配度而不仅是能力 |
| 组织价值 | 招聘准确率提升，新人留存率提高，减少"入错人"的成本 |

**设计启示**：很多培训产品只做到"技能习得层"——教方法、给工具。DDI 的设计强在**三层都明确**，特别是"组织价值层"——这让 HR 在向业务部门推荐课程时，不是用"这门课很好"，而是用"这门课能解决你招错人的问题"。

## 四、核心设计概念

以下是从 DDI 体系中萃取的关键设计概念，不仅仅是方法，更是**产品设计的底层思考**。

### 4.1 行为模式法 —— DDI 最核心的设计概念

**定义**：将任何一项领导力能力拆解为一组**可观测、可练习、可评估的关键行动**。

**不是**：讲理论、讲案例、讲别人的故事。
**而是**：拆成动作 → 练 → 反馈 → 再练。

**为什么这是核心壁垒？**

理论谁都可以讲，但把一个抽象能力（如"影响力"）拆成"了解立场→选择策略→包装信息→获得承诺→评估跟进"这一串具体动作，需要的是**对管理行为的深度研究和大量实证数据**。这是 DDI 50年积累的核心资产。

**对设计的要求**：每开发一门新课，不是先问"这门课要讲什么内容"，而是先问"做好这件事，需要哪几个关键行动？每个关键行动的标准是什么？"

### 4.2 准备度概念 —— 能力不能在真空中建立

**定义**：学员在学习具体管理技能之前，必须先具备三个条件：
- 知道自己是谁（自我认知）
- 知道管理者该做什么（角色认知）
- 知道自己的差距在哪（能力认知）

**DDI 的隐含假设**：一个对自己没有清晰认知的管理者，学再多方法也用不出来。因为管理不是技术活，是**人格+角色+能力**的整合输出。

**这解释了为什么 DNA 测评是"准备课程"而非"附加服务"**——它是整个体系的前置条件。

### 4.3 最小可训单位 —— 模块化设计哲学

**定义**：每项领导力能力是一个独立的、完整的、可单独交付的培训单位。

**设计含义**：
- 不是"领导力培训"这个大产品，而是"16个独立的能力胶囊"
- 每个胶囊有自己的目标、方法、模拟、考核
- 企业可以按需组合，不必买整套
- 但每个胶囊之间有内在的学习路径（先辅导再授权，先建团队再驱动变革）

**这种设计的商业价值**：降低购买门槛（可以只买一门），提高续购率（学完一门自然需要下一门），保持产品灵活性（不同客户不同组合）。

### 4.4 认知缺口驱动 —— DDI 的学习动机设计

**定义**：DDI 的四步闭环（诊断→学习→模拟→考核），第一步不是"教"，而是**让学员发现自己不会**。

**设计逻辑**：
```
诊断（前测）→ "原来我这方面只有2分" → 创造学习动机
         ↓
微课学习 → "原来可以这样做" → 满足认知需求
         ↓
情景模拟 → "让我试试看" → 安全环境中应用
         ↓
能力考核 → "我真的进步了" → 行为改变被确认
```

**这与传统培训的根本不同**：传统培训假设学员"知道自己需要学"，DDI 的设计假设学员"不知道自己哪里不够好"。诊断不是为了评测，而是为了**创造学习动机**。

### 4.5 行为改变的可视化 —— 培训效果的被看见

**定义**：DDI 不只是让学员改变，而是**让改变被看见**——被学员自己看见，被 HR 看见，被业务 leader 看见。

**实现方式**：前测和后测的数据对比。"学习前你的决策能力评分是2.3，学习后是4.1"。

**这是一个关键的商业设计**：企业培训最大的敌人不是"没效果"，而是"效果看不见"。当 HR 拿不出数据证明培训有用时，明年的预算就没了。DDI 的前后测对比直接解决了这个信任问题。

### 4.6 痛点即入口 —— 课程定位的内容策略

**定义**：DDI 每门课不是从"课程内容"开始介绍，而是从**目标学员正在经历的痛苦**开始。

**他们用的句式结构**：
1. 先说痛点（"管理者50%的时间用在沟通上，而50%的问题是由于沟通障碍引起的"）
2. 再说后果（"缺乏必要的互动技巧，需要协同时处处受阻"）
3. 然后用一句话承诺（"七大关键行动，无往不胜的管理沟通心法"）
4. 最后给课程内容

**设计启示**：课程介绍的顺序不是"我们有什么"→"你为什么需要"，而是"你正在痛苦什么"→"我们恰好有这个解决方案"。这是**用户思维和产品思维的差异**。

```
│            ↓                                       │
│  [出口]  行为改变可视化 → 组织价值被验证              │
│                                                    │
│  [核心假设]                                        │
│  1. 领导力是可拆解、可训练、可衡量的行为              │

│  3. 最小的培训单位=一项独立能力                      │
│  4. 学习动机来自对自我缺口的认知，而非外部要求         │
│  5. 培训效果必须被数据证明，而非被感知                │
│                                                    │
└──────────────────────────────────────────────────┘
```

## 六、对摩根课程体系设计的启示

### 6.1 四个必须回答的设计问题

DDI 的体系之所以完整，是因为他们在设计之前回答了四个问题。摩根要建立自己的课程体系，也需要先回答这四个问题：

| DDI 的回答 | 摩根需要回答的对应问题 |
|-----------|---------------------|
| 基中层管理者从"自己做"到"带人做"需要跨越什么？ | **出海企业的管理者从"国内管"到"跨国管"需要跨越什么？** |
| 领导力=一组可观测的关键行动 | **跨文化领导力/出海管理=哪一组可观测的关键行动？** |
| 能力诊断→微课→模拟→考核 | **摩根的课程应该用什么样的统一教学范式？** |
| 大五测评作为自我认知的入口 | **跨文化适应力测评能否作为摩根课程体系的入口？** |

### 6.2 DDI 可迁移的设计概念

| 概念 | DDI 做法 | 摩根应用方向 |
|------|---------|------------|
| 行为模式法 | 每项能力拆成关键行动 | 将"跨文化沟通""海外团队管理"等拆成关键行动序列 |
| 准备度 | DNA测评前置 | 出海准备度测评前置（文化适应力+语言基线+管理成熟度） |
| 递进模型 | 自我→业务→团队→人际 | 国内准备→出海过渡→驻外管理→全球化领导 |
| 能力胶囊 | 16门独立可组合课程 | 将出海培训拆成可组合的能力单元 |
| 认知缺口驱动 | 前测创造学习动机 | 出海模拟前测（让学员在模拟中"失败"，创造学习动力） |
| 改变可视化 | 前后测数据对比 | 出海能力的前后评估数据对比（语言+文化+管理） |
| 痛点即入口 | 课程介绍从痛点开始 | 所有方案从客户的出海失败案例/代价开始 |

### 6.3 关键认知

**DDI 的真正壁垒不是有19门课，而是有一套"把任何管理能力变成可训练的行为"的方法论和生产流程。**

摩根不需要19门课。摩根需要的是一套自己的**行为拆解方法论**——针对"出海管理者"这个特定人群，定义：
1. 他们需要跨越什么？（递进模型）
2. 每层需要哪几个关键行为？（行为拆解）
3. 如何让他们在学习之前先意识到自己"不会"？（认知缺口）
4. 如何让改变可以被数据证明？（可视化）

回答了这四个问题，课程体系的骨架就立起来了。具体开发几门课、每门课讲什么，只是在这个骨架上填肉。
